# PROGRESS — 이미지 분류 에이전트 (`_docs/image-classifier-agent.md`)

형식: `Phase명 / 상태 / DoD 통과 여부 / 특이사항`

- **Phase 1 — 모델 서빙 레이어** / 완료 / DoD 4개 항목 전부 통과 / `timm.create_model("convnext_nano", pretrained=True)` → `convnext_nano.in12k_ft_in1k` 태그로 자동 해석됨(1000-class, 224x224). `torch.onnx.export`는 `dynamo=True`(기본값)일 때 62MB 가중치를 `.onnx.data`로 분리 저장하길래 `dynamo=False`(레거시 익스포터)로 단일 파일(62.4MB) 생성하도록 변경. `onnxruntime-gpu` 최신(1.27.0)은 CUDA 13을 요구해 설치된 torch(cu126)와 충돌 → `1.24.4`로 다운그레이드해 해결. **중요**: `onnxruntime-gpu`가 CUDA를 실제로 쓰려면 `LD_LIBRARY_PATH`에 `site-packages/nvidia/*/lib` 경로들을 넣어줘야 함(torch는 내부적으로 처리하지만 onnxruntime은 별도) — Phase 4(Docker)에서 반영 필요. GPU 첫 추론은 CUDA 컨텍스트 초기화 때문에 ~1000ms로 CPU(~20ms)보다 느리게 로그되는데, 이는 웜업 비용이라 정상이다(반복 호출 시 수 ms대로 떨어짐 — Phase 2/4에서 세션을 프로세스 생애주기 동안 재사용하므로 실사용에서는 문제 없음).
- **Phase 2 — FastAPI `/classify` 엔드포인트** / 완료 / DoD 4개 항목 전부 통과 / Phase 1의 `image_path` 기반 인터랙터를 `content: bytes` + async로 리팩터링(실제 HTTP 업로드에 연결된 선례는 `FaceDetectionInteractor`가 아니라 `FaceRecognitionInteractor`였음 — 전자는 어디서도 호출되지 않는 죽은 코드). `POST /api/vision/classify`로 마운트(기존 `vision_router` prefix 재사용). `dependencies/image_classifier_provider.py`는 다른 provider들과 달리 모듈 전역 싱글턴으로 캐시함 — ONNX 세션 로딩이 요청당 수백ms~1s라 매 요청 재생성하면 응답이 느려짐(이 저장소의 기존 provider들은 전부 요청마다 재생성하지만, 그 인터랙터들은 실제로 안 쓰이거나 훨씬 가벼움). `CLASSIFIER_CONFIDENCE_THRESHOLD` 환경변수로 override 가능. `core/config.py`는 기존에 어디서도 import되지 않는 죽은 모듈이었는데(DATABASE_URL도 database.py가 직접 os.getenv), 새 상수를 여기 추가했지만 라우터는 안전하게 `os.getenv`를 직접 읽음(죽은 경로에 의존하지 않기 위해).
- **Phase 3 — MCP `classify_image` 툴** / 완료 / DoD 3개 항목 전부 통과(단, "MCP inspector로 호출" 대신 tool 함수를 직접 호출하는 방식으로 검증 — 아래 특이사항 참고) / `apps/ontology/adapter/inbound/mcp/image_classifier_tools.py`에 `FastMCP("ImageClassifier")` + `classify_image` tool 작성, `main.py`의 기존 `_SILICON_VALLEY_MCP_SERVERS` 마운트 메커니즘을 재사용해 `_ONTOLOGY_MCP_SERVERS`로 `/mcp/classifier`에 별도 마운트(이름 그대로 캐릭터 롤플레이용 튜플에 섞지 않음). 내부적으로 `/api/vision/classify`를 httpx로 호출(입력은 로컬 경로 또는 URL). 처음엔 파일을 `application/octet-stream`으로 보내 라우터의 이미지 타입 검증(422)에 걸렸다 → `mimetypes.guess_type`으로 실제 mime type 추정하도록 수정. URL 다운로드는 User-Agent 없이 위키미디어에 403을 맞아 UA 헤더 추가. **검증 방법 관련 특이사항**: MCP Python SDK의 `@mcp.tool()`은 이 버전에서 데코레이터가 원본 함수를 그대로 반환해 `classify_image(...)`로 직접 호출 가능 — 실제 MCP JSON-RPC 클라이언트(mcp inspector 등)를 세워 프로토콜 왕복까지 하진 않았고, tool의 실질 로직(로컬 경로/URL 입력, 정상 분류, 존재하지 않는 경로, 분류 서비스 다운 시 에러 메시지)을 직접 호출로 확인했다. main.py 기동 시 세션 매니저가 5→6개로 늘어난 것으로 마운트 자체는 확인됨.
- **Phase 4 — 배포/인프라 통합** / 완료(단, GPU 패스스루 자체는 이 환경에서 검증 불가 — 아래 참고) / DoD 4개 항목 중 3개 실행 검증, 1개(GPU 인식)는 이 환경의 구조적 한계로 "폴백 동작 확인"으로 대체 / 기존 `Dockerfile`은 그대로 두고(이미 이 앱을 포함한 단일 이미지 — 새 Dockerfile 만들지 않음, 근거는 README/배포 문서 참고), `requirements.txt`에 `prometheus-fastapi-instrumentator` 추가, `main.py`에 `Instrumentator().instrument(app).expose(app)`로 `/metrics` 노출. `docker_entrypoint.py`에 `site.getsitepackages()` 기반으로 nvidia pip 패키지의 `lib/` 경로를 모아 `LD_LIBRARY_PATH`를 컨테이너 기동 시 자동 설정하는 로직 추가(Phase 1에서 발견한 onnxruntime-gpu 요구사항). Cloudflare Tunnel은 새 라우팅이 필요 없음(기존 `api.pmhllll12.cloud`→`backend:8000` 라우트가 새 경로도 그대로 받음)을 확인해 `_docs/image-classifier-deployment.md`에 문서화 — 실제 tunnel 설정은 건드리지 않음. **실제로 `docker build`(python:3.13-slim, 로컬 dev 3.14와 다른 버전)를 끝까지 돌려서 검증**: 컨테이너 기동 → 컨테이너 안에서 `export_onnx.py` 실행(62.4MB 생성, 로컬과 동일) → `/classify` 호출 결과가 로컬과 완전히 동일(minibus 69.9%) → `/metrics`에 `/api/vision/classify` 핸들러별 duration/size 카운터 정상 노출. **GPU 관련**: `docker run --gpus all`을 시도했더니 이 환경의 Docker 데몬 자체가 "no known GPU vendor found"로 즉시 거부 — NVIDIA Container Toolkit이 이 환경엔 아예 설치돼 있지 않다는 뜻(추측이 아니라 직접 확인됨). `--gpus` 없이 실행했을 땐 onnxruntime이 CUDA 초기화 실패(`CUDA driver version is insufficient`)를 로그로 남기고 자동으로 CPUExecutionProvider로 폴백해 정상 동작 — 크래시 없음을 확인했다. 실제 프로덕션 서버에 GPU 패스스루를 붙이는 건 호스트 설정이 필요해 이 저장소 변경만으로는 검증도 적용도 불가능하다(`_docs/image-classifier-deployment.md`에 필요 조치 문서화).
- **Phase 5 — 검증 및 문서화** / 완료 / DoD 3개 항목 전부 통과 / 단위 테스트(`tests/app/use_cases/test_image_classifier_interactor.py`: softmax·전처리 예외·threshold·결정론성) + 어댑터 테스트(`tests/adapter/outbound/.../test_image_classifier_model_adapter.py`) + 통합 테스트(`tests/adapter/inbound/api/v1/test_image_classifier_router.py`: FastAPI `TestClient`, 라우터만 붙인 격리된 앱 + 스텁 유스케이스로 검증 로직 테스트, 실모델로 end-to-end 1건) 총 12개 전부 통과(`python -m pytest apps/ontology/tests/`, 기존 21개 포함 총 26개 전부 통과 확인). 외부 샘플 이미지(ultralytics 번들)에 처음 의존했다가 (1) 이 worktree엔 `.venv`가 없어 경로가 깨지고 (2) 저작권 있는 사진을 저장소에 넣는 것도 부적절해서 → PIL로 즉석에서 만드는 합성 이미지(단색/그라디언트/노이즈)로 전부 교체, 외부 파일 의존 없이 실제 ONNX 모델을 그대로 태우는 방식으로 바꿈. `scripts/classifier_scenario_test.py`: 합성 이미지 13종(정상 5 + 애매 5 + 손상 3) + 로컬에 ultralytics 있으면 보너스 실사진 2장, 전부 사람이 읽을 수 있는 결과 출력 — 실행 확인함(체커보드가 "crossword puzzle" 64.7%로 나오는 등 재미있는 결과 포함). `README.md`(`apps/ontology/README.md`)에 아키텍처 다이어그램(텍스트), 실행법, API 스펙, MCP 등록법, 알려진 한계점(ImageNet-1k 한정/요청당 latency/GPU 패스스루 별도 설정 필요/MCP 프로토콜 왕복 미검증/import-linter 미적용) 전부 명시. `pytest.ini`의 `testpaths`에 ontology가 원래 빠져있는 기존 문제는 손대지 않음(이미 `_docs/CLAUDE.md`에 우회 방법이 문서화돼 있는 기존 상태) — `python -m pytest apps/ontology/tests/ -v`로 명시 실행 필요.
