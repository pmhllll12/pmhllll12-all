# PROGRESS — 이미지 분류 에이전트 (`_docs/image-classifier-agent.md`)

형식: `Phase명 / 상태 / DoD 통과 여부 / 특이사항`

- **Phase 1 — 모델 서빙 레이어** / 완료 / DoD 4개 항목 전부 통과 / `timm.create_model("convnext_nano", pretrained=True)` → `convnext_nano.in12k_ft_in1k` 태그로 자동 해석됨(1000-class, 224x224). `torch.onnx.export`는 `dynamo=True`(기본값)일 때 62MB 가중치를 `.onnx.data`로 분리 저장하길래 `dynamo=False`(레거시 익스포터)로 단일 파일(62.4MB) 생성하도록 변경. `onnxruntime-gpu` 최신(1.27.0)은 CUDA 13을 요구해 설치된 torch(cu126)와 충돌 → `1.24.4`로 다운그레이드해 해결. **중요**: `onnxruntime-gpu`가 CUDA를 실제로 쓰려면 `LD_LIBRARY_PATH`에 `site-packages/nvidia/*/lib` 경로들을 넣어줘야 함(torch는 내부적으로 처리하지만 onnxruntime은 별도) — Phase 4(Docker)에서 반영 필요. GPU 첫 추론은 CUDA 컨텍스트 초기화 때문에 ~1000ms로 CPU(~20ms)보다 느리게 로그되는데, 이는 웜업 비용이라 정상이다(반복 호출 시 수 ms대로 떨어짐 — Phase 2/4에서 세션을 프로세스 생애주기 동안 재사용하므로 실사용에서는 문제 없음).
- **Phase 2 — FastAPI `/classify` 엔드포인트** / 완료 / DoD 4개 항목 전부 통과 / Phase 1의 `image_path` 기반 인터랙터를 `content: bytes` + async로 리팩터링(실제 HTTP 업로드에 연결된 선례는 `FaceDetectionInteractor`가 아니라 `FaceRecognitionInteractor`였음 — 전자는 어디서도 호출되지 않는 죽은 코드). `POST /api/vision/classify`로 마운트(기존 `vision_router` prefix 재사용). `dependencies/image_classifier_provider.py`는 다른 provider들과 달리 모듈 전역 싱글턴으로 캐시함 — ONNX 세션 로딩이 요청당 수백ms~1s라 매 요청 재생성하면 응답이 느려짐(이 저장소의 기존 provider들은 전부 요청마다 재생성하지만, 그 인터랙터들은 실제로 안 쓰이거나 훨씬 가벼움). `CLASSIFIER_CONFIDENCE_THRESHOLD` 환경변수로 override 가능. `core/config.py`는 기존에 어디서도 import되지 않는 죽은 모듈이었는데(DATABASE_URL도 database.py가 직접 os.getenv), 새 상수를 여기 추가했지만 라우터는 안전하게 `os.getenv`를 직접 읽음(죽은 경로에 의존하지 않기 위해).
- **Phase 3 — MCP `classify_image` 툴** / 완료 / DoD 3개 항목 전부 통과(단, "MCP inspector로 호출" 대신 tool 함수를 직접 호출하는 방식으로 검증 — 아래 특이사항 참고) / `apps/ontology/adapter/inbound/mcp/image_classifier_tools.py`에 `FastMCP("ImageClassifier")` + `classify_image` tool 작성, `main.py`의 기존 `_SILICON_VALLEY_MCP_SERVERS` 마운트 메커니즘을 재사용해 `_ONTOLOGY_MCP_SERVERS`로 `/mcp/classifier`에 별도 마운트(이름 그대로 캐릭터 롤플레이용 튜플에 섞지 않음). 내부적으로 `/api/vision/classify`를 httpx로 호출(입력은 로컬 경로 또는 URL). 처음엔 파일을 `application/octet-stream`으로 보내 라우터의 이미지 타입 검증(422)에 걸렸다 → `mimetypes.guess_type`으로 실제 mime type 추정하도록 수정. URL 다운로드는 User-Agent 없이 위키미디어에 403을 맞아 UA 헤더 추가. **검증 방법 관련 특이사항**: MCP Python SDK의 `@mcp.tool()`은 이 버전에서 데코레이터가 원본 함수를 그대로 반환해 `classify_image(...)`로 직접 호출 가능 — 실제 MCP JSON-RPC 클라이언트(mcp inspector 등)를 세워 프로토콜 왕복까지 하진 않았고, tool의 실질 로직(로컬 경로/URL 입력, 정상 분류, 존재하지 않는 경로, 분류 서비스 다운 시 에러 메시지)을 직접 호출로 확인했다. main.py 기동 시 세션 매니저가 5→6개로 늘어난 것으로 마운트 자체는 확인됨.
